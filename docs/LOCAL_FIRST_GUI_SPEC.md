# ShipReady Local-First GUI Spec

## 1. Product Goal

ShipReady's first GUI should turn the proven local CLI engine into a guided, understandable workflow for people who have generated a website and want to know whether it is ready to share, index, and deploy.

The core promise:

> Your generated website is almost ready. Here is what the internet sees, what is missing, and what ShipReady can safely fix.

The first UI should be a guided flow, not a dense audit dashboard. It should help the user move through:

```txt
URL -> local repo -> audit -> explain issues -> plan fixes -> preview changes -> apply safe fixes -> re-check
```

The GUI should make abstract metadata tangible by showing what search engines, social apps, and preview bots can see. It should avoid SEO-cockpit language, ranking promises, and panic-inducing scores.

## 2. Target Users

Primary users:

- Non-technical or semi-technical founders with websites generated by Lovable, v0, Claude, ChatGPT, Cursor, Bolt, Replit, or similar tools.
- Designers, marketers, consultants, and indie builders who know their website URL and may know where the local project folder is.
- Technical users who want a safer visual wrapper around the CLI before changing files.

They likely know:

- Their public website URL.
- Where the local project or exported source folder might be.
- Whether they deployed through Vercel, Netlify, Lovable, Replit, or another host.

They likely do not understand:

- Canonical URLs.
- Open Graph tags.
- Twitter/X card metadata.
- JSON-LD.
- Raw HTML versus rendered HTML.
- robots.txt and sitemap.xml conventions.
- Framework-specific metadata surfaces.
- Why metadata injected by JavaScript can be missed by some preview bots.

## 3. Non-Goals

The first GUI MVP should not include:

- Account system.
- Billing.
- Multi-user workspaces.
- GitHub OAuth.
- Pull request creation.
- Deployment.
- Monitoring.
- Keyword research.
- Backlink analysis.
- Content planning.
- Full SEO strategy.
- CMS integrations.
- Competitor analysis.
- GUI-driven crawl exploration beyond the current report flow. Pass 15's `crawl` CLI/MCP contract provides a bounded same-origin sample, but wiring that contract into the GUI is a future GUI revisit.
- Automated metadata, JSON-LD, content, accessibility text, package, config, Git, commit, or deploy writes.
- A general-purpose expert SEO dashboard.

## 4. Primary User Journey

Recommendation: guided flow first.

1. The user enters a website URL.
2. The user selects a local project folder, or chooses URL-only audit.
3. ShipReady audits what the live internet sees.
4. ShipReady explains top issues in plain language with visual previews.
5. ShipReady inspects the local repo and explains what kind of project it appears to be.
6. ShipReady generates a fix plan grouped by safety and review level.
7. ShipReady previews the exact patch.
8. ShipReady applies only safe V1 file creations when eligible.
9. ShipReady re-checks locally and explains that the live website will not change until the user deploys.

The ideal first success moment:

```txt
Your site is now safer to crawl.

ShipReady created:
- app/robots.ts
- app/sitemap.ts

Next:
- Review the created files.
- Run your project tests.
- Deploy when ready.
- Re-run ShipReady after deployment.
```

The UI must not imply that ShipReady deployed the site or that the live URL changed immediately after local files were created.

## 5. Recommended Information Architecture

Use a step-based application shell:

- Left stepper: Connect, Readiness, Project, Fix plan, Preview, Apply, Re-check.
- Central pane: the current task, preview, or result.
- Right panel: safety notes, plain-language explanation, and collapsible developer details.
- Bottom action bar: primary next action and secondary back/cancel/export actions.

Primary navigation should keep the user oriented without turning the app into a dashboard. The user should always know:

- What ShipReady checked.
- What is missing.
- What can be safely applied.
- What needs human review.
- Whether anything has changed on disk.
- Whether the live deployed site has changed.

Advanced details should be present but collapsed:

- Raw check IDs.
- JSON output.
- SHA-256 hashes.
- Exact metadata tag names.
- Full diff hunks.
- Framework internals.
- Raw and rendered extraction data.

## 6. Screen-by-Screen UX Spec

### Screen 1 - Start / Connect

Purpose: collect the minimum information needed to start.

Primary copy:

```txt
Check if your generated website is ready to be shared and indexed.
```

Inputs:

- Website URL.
- Local project folder.
- Optional "Audit URL only" path.

Primary CTA:

```txt
Check my site
```

Secondary CTA:

```txt
Audit URL only
```

Expected behavior:

- URL is required for all paths.
- Folder is required only for fix planning and applying safe fixes.
- If the user chooses URL-only audit, the flow skips repo understanding, fix plan, patch preview, and apply.
- If a folder is selected, run repo inspection before or alongside the audit so the next screens can show framework confidence.

States:

- Empty: show two simple inputs and no technical explanation.
- Invalid URL: "Enter a full website address, like https://example.com."
- Inaccessible URL: "ShipReady could not reach this page. Check that it is public and deployed."
- Folder not selected: "You can still audit the live URL, but ShipReady needs the project folder before it can plan safe local fixes."
- Unsupported folder: "This folder does not look like a supported website project yet. You can continue with a URL-only audit."
- Loading: show "Checking what the internet sees..." and avoid exposing command names.

Data source:

- `audit <url>` or `audit <url> --json`.
- `inspect-repo <path>` or `inspect-repo <path> --json` when a folder is selected.
- CLI command errors for invalid URL, inaccessible URL, and invalid repo path.

### Screen 2 - Site Readiness Overview

Purpose: summarize what the live internet sees and make metadata visible.

Main status:

- Prefer a readiness label over a large numeric score.
- Use "Ready", "Almost ready", or "Needs attention".
- Keep the raw `score` available in details, not as the main emotional anchor.

Core sections:

- Top issues, limited to the 3-5 most important.
- What passed.
- Visual preview cards.
- "What preview bots see" raw-versus-rendered summary.
- Continue button to review the local project and fix plan.

Recommended preview cards:

- Google search preview: title, URL, description.
- LinkedIn/social preview: image, title, description, URL.
- X/Twitter preview: card type, image, title, description.
- Slack/Discord preview: image, title, description, URL.
- What crawlers see: raw HTML versus rendered HTML.

Copy examples:

- "Search engines do not have a clear preferred URL for this page."
- "Your share image is missing, so social apps may show a plain link."
- "Some page information appears only after the app loads. Some preview bots may not see it."

States:

- Good result: "No launch-blocking metadata or crawlability issues were found."
- Needs attention: "A few important details are missing before this page is ready to share widely."
- Critical live issue: "This page may not be reachable or indexable yet."
- Render-only warning: show a visible "Some bots may miss this" callout.
- URL-only mode: show "Select a local project folder to preview safe fixes."

Data source:

- `AuditResult.status`, `score`, `checks`, `resources`, `raw`, `rendered`, and `comparison`.
- `raw.metadata` for crawler-visible previews.
- `rendered.metadata` for "after app loads" comparisons.
- `resources.robotsTxt` and `resources.sitemapXml` for crawl resource status.
- Pass 13's `social-preview` CLI/MCP contract provides a richer read-only simulator, but wiring that contract into the GUI is a future GUI revisit rather than part of Pass 13.

### Screen 3 - Repo Understanding

Purpose: show whether the selected local folder matches the live site well enough to plan safe changes.

Content:

- Detected framework.
- Confidence.
- Package manager.
- Important files.
- Likely routes/pages.
- Likely metadata locations.
- Supported future fixes.
- Limitations and warnings.

Plain-language framework copy:

- "This looks like a Next.js App Router project."
- "This looks like a Vite React project."
- "This looks like a static HTML site."
- "ShipReady is not confident about this folder yet."

Confidence model:

- High: "Good match for safe planning."
- Medium: "Likely match, but review carefully."
- Low: "Manual review recommended."

Unsupported/manual-only state:

```txt
ShipReady can audit the live URL, but this folder is not supported for safe local fixes yet.
```

Data source:

- `RepoInspectionResult.framework`.
- `RepoInspectionResult.packageManager`.
- `RepoInspectionResult.importantFiles`.
- `RepoInspectionResult.routes`.
- `RepoInspectionResult.metadataLocations`.
- `RepoInspectionResult.supportedFixes`.
- `RepoInspectionResult.limitations`.
- `RepoInspectionResult.warnings`.

### Screen 4 - Fix Plan

Purpose: translate audit findings and repo inspection into grouped actions.

Groups:

- Safe to apply.
- Needs review.
- Manual only.
- Already good.
- Optional polish.

Each action should show:

- Plain-language title.
- Why it matters.
- Target file or location.
- Risk/review status.
- Whether ShipReady can apply it in V1.
- Whether it affects the live deployed site only after deployment.

Recommended display rules:

- Do not show "critical" as the first label for non-technical users.
- Do not present JSON-LD missing as blocking by default.
- Safe to apply should be reserved for creation-only robots/sitemap candidates that can pass V1 write policy.
- Metadata, JSON-LD, H1, alt text, package, config, Git, and deploy changes must show as preview-only, review-required, manual-only, or out of scope.

Data source:

- `FixPlanResult.auditSummary`.
- `FixPlanResult.repoSummary`.
- `FixPlanResult.actions`.
- `FixPlanResult.noActionChecks`.
- `FixPlanResult.optionalNotes`.
- `FixPlanResult.limitations`.
- `FixPlanResult.recommendedNextStep`.

Current engine mapping:

- `safe_automated_later` -> UI "Safe to apply" only if the later dry-run/write result confirms V1 eligibility. Otherwise label "Safe candidate".
- `automated_with_review` -> UI "Needs review".
- `manual_recommendation` -> UI "Manual only".
- `noActionChecks` -> UI "Already good".
- `optionalNotes` -> UI "Optional polish".

### Screen 5 - Patch Preview

Purpose: show exactly what would happen before anything changes.

Core copy:

```txt
No files have been changed yet.
```

Content:

- Files ShipReady would create.
- Files ShipReady would update, if any, clearly marked review-required and not writable in V1.
- Skipped or blocked actions.
- Diff preview.
- Safety notes.
- "Apply safe fixes" button only when eligible write candidates exist.

Apply button rules:

- Enable only when at least one dry-run file change is an eligible V1 creation candidate.
- The candidate must be a `create` change.
- The candidate must be low risk.
- `requiresHumanReview` must be false.
- `reviewStatus` must be `auto_candidate`.
- The target path must be in the V1 robots/sitemap allowlist.
- Updates must not be included in the apply action.

Recommended safety callouts:

- "ShipReady will only create missing robots/sitemap files."
- "ShipReady will not overwrite existing files."
- "ShipReady will not edit metadata or page content."
- "ShipReady will not commit, push, or deploy."

Data source:

- `DryRunFixResult.fileChanges`.
- `DryRunFixResult.skippedActions`.
- `DryRunFixResult.safetyNotes`.
- `DryRunFixResult.recommendedNextStep`.
- `DryRunFileChange.diff`, `before`, and `after` for advanced preview.

### Screen 6 - Apply Result

Purpose: report exactly what changed and what did not.

Success content:

- Files created.
- Files actually changed count.
- Blocked changes.
- Skipped actions.
- Safety confirmations.
- Recommended next steps.

Primary success copy:

```txt
Your site is now safer to crawl locally.
```

Required deployment warning:

```txt
These files were created in your local project. Your live website will not change until you deploy.
```

When no files were written:

```txt
ShipReady did not change any files.
```

Safety statements:

- No existing files overwritten.
- No files outside the repo root touched.
- No Git operations performed.
- No commits created.
- No dependencies installed.
- No package files or lockfiles changed.
- No formatting run.
- No deploys performed.

Advanced details:

- Byte count and SHA-256 for created files.
- Safety checks.
- Rollback result if present.

Data source:

- `WriteFixResult.wroteFiles`.
- `WriteFixResult.policy`.
- `WriteFixResult.createdFiles`.
- `WriteFixResult.blockedChanges`.
- `WriteFixResult.skippedActions`.
- `WriteFixResult.safetyChecks`.
- `WriteFixResult.rollback`.
- `WriteFixResult.recommendedNextStep`.

### Screen 7 - Re-check

Purpose: separate local improvement from live deployed state.

The UI should show two lanes:

- Local project check: "ShipReady no longer sees those missing local files."
- Live website check: "The deployed site still changes only after you deploy."

Recommended states:

- After local write, before deployment:

```txt
Local files are ready. Deploy your project to update the live website.
```

- After deployment and re-audit:

```txt
The live website now exposes the expected crawl files.
```

- If live audit still reports missing robots/sitemap:

```txt
The live URL does not show these changes yet. Deploy the project, then run the live check again.
```

Data source:

- Local re-check can run `inspect-repo <path>` and `fix <path> --url <url> --dry-run` to confirm there are no remaining eligible creation candidates.
- Live re-check uses `audit <url>`.
- The current engine does not audit local source output or preview an undeployed build. The UI must make that limitation clear.

## 8. Non-Technical Vocabulary And Copy Guidelines

Writing principles:

- Lead with the user outcome, not the technical artifact.
- Explain why something matters in one sentence.
- Avoid ranking promises.
- Avoid jargon unless hidden under "For developers".
- Use calm labels. Do not make ordinary metadata gaps feel catastrophic.
- Reserve scary language for unreachable, noindex, blocked-by-robots, or clear live crawl failures.

Recommended translations:

| Technical term | User-facing copy |
| --- | --- |
| Missing canonical URL | Search engines do not have a clear preferred URL for this page. |
| Invalid sitemap | Your sitemap address exists, but it returns a normal app page instead of a sitemap file. |
| Metadata appears only after JavaScript rendering | Some page information appears only after the app loads. Some preview bots may not see it. |
| JSON-LD missing | Your site does not yet describe itself in structured data for search and AI tools. |
| Missing og:image | Your share image is missing, so social apps may show a plain link. |
| robots.txt missing | Your site does not have a simple crawl instructions file yet. |
| robots.txt blocks page | Your crawl instructions may be telling search engines not to visit this page. |
| noindex | This page is asking search engines not to index it. |
| H1 missing | The page does not have one clear main heading. |
| Image alt missing | Some images are missing text descriptions. |
| Raw HTML | What many crawlers and preview bots receive first. |
| Rendered HTML | What appears after the app loads in a browser. |

Recommended user-facing labels:

- Ready.
- Almost ready.
- Needs attention.
- Safe to apply.
- Needs review.
- Manual only.
- Already good.
- Optional polish.
- Preview only.
- Local change.
- Live site unchanged until deploy.
- For developers.

Avoid as primary UI labels:

- Critical.
- SERP.
- Canonical.
- Open Graph.
- JSON-LD.
- SSR.
- CSR.
- Hydration.
- Meta tags.
- Robots directive.
- Patch hunk.

These terms can appear inside expanded developer details.

## 9. Severity Model For UI

Use two separate concepts:

1. Outcome severity: how much the issue affects readiness.
2. Fix safety: whether ShipReady can apply the fix.

Outcome severity:

- Blocking: the live page may not be reachable, indexable, or visible to bots. Examples: non-200 response, noindex, robots blocking the page, title and description only appearing after JavaScript.
- Important: the site can load, but important preview or crawler signals are missing or inconsistent. Examples: missing title, missing description, missing canonical, missing og:image, invalid sitemap, sitemap missing audited URL.
- Recommended: useful completeness work that improves interpretation but is not a launch blocker. Examples: missing JSON-LD, missing Twitter fields, missing language, missing favicon, weak title/description.
- Optional polish: nice-to-have visual or browser polish. Example: theme color.
- Ready: passed checks.

Fix safety:

- Safe to apply: V1-eligible creation-only robots/sitemap changes that pass dry-run and write safety gates.
- Needs review: metadata, JSON-LD, sitemap updates, content semantics, alt text, and framework-specific changes that may be previewed but should be reviewed.
- Manual only: unsupported framework, rendering architecture changes, inaccessible deployment, Git/deploy decisions, or anything requiring business facts.
- Already good: passed checks.
- Preview only: changes ShipReady can explain or preview but must not write in V1.

Mapping from current engine:

- `AuditCheck.severity === "passed"` -> Ready.
- `AuditCheck.severity === "info"` -> Optional polish.
- `AuditCheck.severity === "warning"` -> Important or Recommended, depending on check ID.
- `AuditCheck.severity === "critical"` -> Blocking or Important, depending on check ID.
- `FixPlanAction.category === "safe_automated_later"` -> Safe candidate, promoted to Safe to apply only after dry-run confirms V1 eligibility.
- `FixPlanAction.category === "automated_with_review"` -> Needs review.
- `FixPlanAction.category === "manual_recommendation"` -> Manual only.

JSON-LD caveat:

The current audit engine can classify missing JSON-LD as `critical`. In the GUI, missing JSON-LD should usually be Recommended, not Blocking, unless a future product mode explicitly focuses on structured data completeness. Invalid JSON-LD can be Important because broken structured data is misleading, but ShipReady should still avoid writing JSON-LD without review.

## 10. Safety Model And Trust Affordances

The GUI must build trust by making file effects explicit before and after every action.

Before preview:

- "ShipReady is reading your live URL and local project."
- "No files have been changed."

Before apply:

- "No files have been changed yet."
- "ShipReady can safely create these missing crawl files."
- "Review-required changes are not included."
- "Existing files will not be overwritten."

During apply:

- Show the exact number of files being created.
- Do not show metadata/content changes as part of the apply action.
- Keep the action label specific: "Create safe crawl files", not "Fix everything".

After apply:

- Show files created.
- Show blocked changes.
- Show skipped actions.
- Show no overwrite, no Git, no commit, no deploy, no dependencies, no formatting.
- Show the local-versus-live deployment warning.

Advanced trust affordances:

- Show byte counts and SHA-256 hashes under "For developers".
- Show the V1 policy name under details: `creation_only_robots_sitemap_v1`.
- Show the dry-run source action IDs under details.
- Show rollback information if present.

V1 safe-write rule:

ShipReady may apply only eligible missing robots/sitemap file creations:

- Static HTML: `robots.txt`, `sitemap.xml`.
- Vite React: `public/robots.txt`, `public/sitemap.xml`.
- Next.js App Router: `app/robots.ts`, `app/sitemap.ts`, `src/app/robots.ts`, `src/app/sitemap.ts`.

Everything else remains review-required, manual-only, blocked, or out of scope.

## 11. Empty, Loading, And Error States

Empty states:

- Start: "Enter your website URL and select your project folder."
- URL-only result: "Add a local project folder when you are ready to preview safe fixes."
- No issues found: "This page looks ready for the checks ShipReady runs today."
- No safe fixes: "ShipReady found things to review, but nothing it should change automatically."
- Unsupported repo: "ShipReady can audit this URL, but this project type is not safe for automated fixes yet."

Loading states:

- Audit: "Checking what the internet sees..."
- Rendered pass: "Opening the page like a browser..."
- Repo inspection: "Reading the project structure..."
- Fix plan: "Matching site issues to local files..."
- Dry run: "Preparing a preview without changing files..."
- Write: "Creating safe crawl files..."
- Re-check: "Checking local files and live URL state..."

Error states:

- Invalid URL: "Enter a full website address, like https://example.com."
- Network error: "ShipReady could not reach this page. Check that it is public and deployed."
- Timeout: "The page took too long to load. Try again, or increase the timeout in developer settings."
- Invalid folder: "This folder could not be inspected. Choose the project root folder."
- Unsupported folder: "This project type is not supported for safe fixes yet."
- Conflicting mode: "Choose either preview or apply, not both."
- Write blocked: "ShipReady did not change files because one or more safety checks failed."
- Permission error: "ShipReady could not create the file. Check folder permissions and try again."

Errors should include a short "For developers" disclosure with the raw CLI error and command stage.

## 14. Suggested Visual Direction

The UI should feel calm, trustworthy, visual, and simple. It should be closer to Linear, Vercel, or Stripe product quality than to an expert SEO cockpit.

Recommended structure:

- Left stepper for flow progress.
- Central result and preview pane.
- Right explanation and safety panel.
- Preview cards for "what the internet sees".
- Expandable developer details.
- Quiet neutral base with restrained accent colors.
- Strong typography hierarchy without oversized marketing hero sections.
- Compact but approachable cards with clear status labels.
- 8px or smaller radius unless a later design system says otherwise.

Visual principles:

- Show the website and previews, not abstract metrics.
- Avoid one giant SEO score.
- Avoid red-heavy panic styling except for true blocking states.
- Use green sparingly for verified ready/safe states.
- Use amber or blue for review/recommended states.
- Keep technical details one click away.
- Make "local files changed" versus "live site changed" visually unmistakable.

## 15. Implementation Options Without Choosing Prematurely

Do not choose implementation technology in this spec. The product can support several paths:

Option A - Local web UI wrapping the CLI:

- Runs a small localhost server.
- Uses the existing CLI engine as a library or subprocess.
- Fastest path to iterate on UX.
- Needs careful local file permissions and process lifecycle design.

Option B - Desktop shell:

- Tauri or Electron style package.
- Stronger folder picker and file system affordances.
- Better "local-first product" feel for non-technical users.
- More packaging and update complexity.

Option C - CLI-generated static report:

- Produces a local HTML report from existing JSON outputs.
- Lowest implementation risk.
- Good stepping stone for preview cards and copy normalization.
- Less interactive for folder picking and safe apply.

Option D - Future SaaS:

- Useful later for monitoring, history, and collaboration.
- Not suitable for first safe local write workflow because file access and repo writes are local.

Recommended implementation posture:

- Keep the engine UI-agnostic.
- Add a UI-oriented normalized report layer before building a rich GUI.
- Let the first GUI consume stable JSON rather than parsing human CLI output.

## 16. Engine And Model Gaps Discovered During Spec

No engine changes are required for this spec, but the GUI would benefit from these model additions:

1. UI-normalized issue model.
   - Existing `AuditCheck.title` and `description` are useful but still technical.
   - Add friendly title, friendly explanation, why-it-matters, user-facing severity, and developer details.

2. Stable UI severity mapping.
   - Current `critical`, `warning`, `info`, `passed` labels are engine-centric.
   - GUI needs Blocking, Important, Recommended, Optional polish, and Ready.
   - Missing JSON-LD should be downshifted in the UI unless the product mode says otherwise.

3. Preview-card normalized data.
   - Existing raw/rendered metadata can power previews, but the GUI needs normalized preview objects for Google, LinkedIn/social, X/Twitter, Slack/Discord, and crawler view.
   - Include selected source, missing fields, fallback behavior, and image URL validation status.

4. Explicit live-versus-local state.
   - Existing commands can infer this, but the GUI needs a first-class `requiresDeploymentToAffectLiveSite` flag on local writes and local dry-run results.

5. Safe apply eligibility.
   - Dry-run has enough primitives to infer eligibility.
   - GUI would be safer with `eligibleForWrite: boolean`, `writePolicy`, and `writeBlockReason` per file change.

6. Unified workflow/session result.
   - Today the GUI must stitch together audit, repo inspection, plan, dry-run, write, and re-check results.
   - A UI report could bundle stages, timestamps, command status, and recovery actions.

7. Friendlier command errors.
   - JSON errors currently expose `ok: false` and `error`.
   - GUI would benefit from stable error codes, stage, retryability, and suggested user action.

8. Git worktree warning.
   - V1 write mode correctly performs no Git operations.
   - A GUI should still warn when the target repo has uncommitted changes, if Git data is available, without blocking safe creations by default.

9. Relationship between live URL and local repo.
   - Repo inspection does not prove the selected folder deploys to the audited URL.
   - GUI should present framework confidence separately from URL-folder confidence until a future model can improve this.

10. Local build/render verification.
    - Current audit checks the live URL only.
    - The GUI cannot truthfully say local source changes fix the live site until deployment and a new live audit.

```txt
Add UI-oriented report normalization to the CLI engine.
```

Why:

- The engine already has the core commands and safe write behavior.
- A GUI should not parse human CLI output.
- The current JSON is technically strong but not shaped around screens, user-facing copy, preview cards, safe apply buttons, or live-versus-local warnings.
- A normalization layer can stay implementation-neutral and support all later UI forms: local web app, desktop shell, static HTML report, or future SaaS.

Suggested output shape:

```txt
shipready ui-report <path> --url <url> --json
```

or an internal library function first:

```txt
createUiReport({ url, repoPath, mode })
```

The normalized report should include:

- Workflow stage summaries.
- User-facing readiness label.
- UI severity mapping.
- Preview-card data.
- Action groups.
- Safe apply eligibility.
- Review-required reasons.
- Local-versus-live deployment warnings.
- Error codes and recovery actions.
- Developer details linked back to existing command results.

Do not implement the GUI until this layer is stable enough for a prototype to consume.
